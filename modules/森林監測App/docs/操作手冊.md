## 壹、平臺簡介

### 一、是什麼

**ForestMRV** 是一個離線可用的 PWA（漸進式網頁應用程式），把森林永久樣區的調查、品管、報告流程數位化。所有資料即時雲端同步，符合 IPCC LULUCF + MRV（測量／報告／查證）原則，可直接做為碳匯認證、森林經營計畫書、永續輔導計畫的資料源。

### 二、能解決什麼問題

| 痛點 | 平臺解法 |
|---|---|
| 紙本調查易遺失、難共享 | 即時雲端同步、離線仍可填、回連自動上傳 |
| 多人共同調查資料品質參差 | 4 種 QA 狀態 + 「待審核」清單 + 標記退回機制 |
| 計畫結束後資料被竄改、無 audit trail | Lock 機制 — 全 verified 後自動進審查並鎖定 |
| 跨計畫看不到全貌 | 儀表板 + 地圖視圖（多模組統計圖） |
| 報告階段才發現座標錯、樹種誤判 | QA 階段就揪出來；surveyor 看「我被 flag」清單即時修 |

### 三、6 大監測模組（v2.2 全到齊）

🌳 **立木調查**　🌱 **自然更新**　🌿 **地被植物**　⛰️ **水土保持**　🦌 **野生動物**　🌰 **經濟收穫**

### 四、瀏覽器與裝置需求

- **桌機**：Chrome 100+ / Edge 100+ / Firefox 100+ / Safari 16+
- **手機**：iPhone Safari 16+ / Android Chrome 100+
- **手機建議「加入主畫面」**：iOS 在 Safari 點分享→「加入主畫面」；Android Chrome 點選單→「加到主畫面」 — 可離線開啟、體驗如原生 App
- **首次使用需網路**（登入 + Service Worker 安裝），之後可離線填表

## 肆、PI 操作流程（建專案 → 結案 完整循環）

### 一、第 1 步：請 admin 建空殼專案

PI 自己**不能**建專案，需聯絡 admin（系統管理者）：
- 提供專案名稱（中文，例如「大雪山林業合作社 2026 永續輔導」）
- 專案代碼（短英數，例如 `DXS-2026`）
- 你的 Google 登入 email（admin 會把你指派為 PI）

### 二、第 2 步：進入專案 → 設定方法學

進入空殼專案後，預設停在「樣區」分頁。

1. 切到「**設計**」分頁
2. 點「**編輯方法學**」
3. 設定以下項目：
   - **目標樣區數**（例如 50）
   - **樣區形狀**（圓形 / 方形）
   - **允許的樣區面積**（逗號分隔，例如 `400, 500, 1000`）— surveyor 加新樣區時只能選這些
   - **強制必填欄位**：照片 / 枝下高 / 病蟲害 三選
   - **啟用模組**：v1 立木 / 自然更新（預設開）；v2 地被 / 水保 / 野生動物 / 經濟收穫（預設關，需要才開）
   - 各模組可有獨立必填設定（例：地被樣方照片必填、水保定點照片必填）
   - **方法學說明**（給其他成員與 reviewer 看的文字）
4. 點「儲存」→ 系統自動把專案狀態從 `created` 升到 `planning`

> 📷 **圖位 3**：方法學編輯表單（顯示樣區形狀、面積、模組勾選區）

### 三、第 3 步：邀請成員

切到「**設定**」分頁 → 「成員與角色」：

1. 在「新成員 email」欄填入對方 Google email
2. 選角色：**調查員**（surveyor）/ **主持人**（pi，多 PI 共治）/ **審查委員**（reviewer）
3. 點「加入」

> ⚠️ 對方必須**先登入過一次**才加得到（否則會失敗）。請先請對方上 https://forestry-rs-monitor.web.app 用 Google 登入一次。

### 四、第 4 步：批量建立空殼樣區（可選但建議）

如果你的方法學已規劃 50 個樣區位置（例：林業署規劃圖），可以一次先建 50 個空殼樣區，後續指派給 surveyor 帶 GPS 去現場填：

1. 「設計」分頁底部，點「＋ 批量建立空殼樣區」
2. 設定：林班（可留空）、起始編號、建立數量、預設面積、預設形狀
3. 預覽顯示「將建立 ABC-001 ～ ABC-050」
4. 點「建立」

建立完到「樣區」分頁可看到 50 個 `qaStatus: shell` 的空殼，再逐一**指派給 surveyor**（卡片右上角下拉）。

## 肆之二、替代方案 — 從 Excel 批次匯入既有資料（v2.6.0 新功能）

> 適用情境：林業保育署永久樣區複測、過去手填 Excel 想一次上雲、紙漿廠等大規模調查（19 樣區 / 930 立木 60-90 秒就匯入完）。
> 取代第 4 步的「批量建空殼樣區 + 逐筆現場填」做法。

### 一、Excel 來源資料規格

| 工作表 | 必要欄位 | 備註 |
|---|---|---|
| 每個樣區一個 sheet | 樣木號碼、樹種中名、DBH、樹高、絕對 X/Y（TWD97）、局部 X/Y、狀態碼、紀錄類型、備註 | sheet 名稱即為樣區編號 |

> 平臺只認**絕對 X/Y（TWD97）**或**局部 X/Y + 一個基準點**（會自動反推樣區中心點）。建議至少留絕對 X/Y 一欄，匯入過程系統會自動處理。

### 二、匯入步驟（PI 入口：「樣區」分頁右上角「📥 從 Excel 匯入」）

**(一) 第 1 步：選檔**

1. 拖曳或點選 .xlsx 檔
2. 系統自動列出所有 sheet — 勾選要匯入的工作表

**(二) 第 2 步：樣區對應**

1. 系統把每個 sheet 名稱當成樣區編號預填
2. 可手動改編號（例：sheet 名稱「測站 1」→ 樣區編號 `CHP-001`）
3. 已存在於專案的編號會標「重複」— 匯入時會自動跳過（idempotent）

**(三) 第 3 步：欄位對應**

1. 系統猜測 Excel 欄名對應到平臺 schema（樣木號碼 → treeNum、DBH → dbh_cm、X → localX_m / absX_twd97...）
2. 必填欄位（樣木號碼、樹種中名、DBH、樹高）若沒對到 → 紅色警示
3. 手動下拉調整

**(四) 第 4 步：樹種與狀態碼比對**

1. 系統把 Excel 樹種中名比對平臺 100 種樹種字典
2. **未知樹種**會列出 — 確認是否拼錯字 / 是否為新樹種
3. 未知樹種仍可繼續匯入：材積/碳量自動套「其他闊」fallback 公式（不擋全批）
4. 狀態碼對照（例：1/2/3 → healthy；5 → standing-dead；6 → fallen）可手動微調

**(五) 第 5 步：DRY-RUN 預覽 → 真實匯入**

1. 預覽區顯示：**樣區數 / 立木數 / 需新增樹種數**三個統計卡
2. 完整 payload JSON 可下載備份（藍色「⬇ 下載 payload.json」按鈕）
3. 確認沒問題 → 點**紅色「✅ 執行真實匯入到 Firestore」**按鈕
4. 確認對話框跳出：列出**樣區數 / 立木數 / 預估時間（每筆約 0.07 秒）/ 未知樹種 fallback 警告**
5. 按「確定」開始寫入：
   - **Phase A 寫樣區**：每筆 addDoc，plot 中心 TWD97 從第一筆 tree 反推（absX − localX）→ 同步換算 WGS84 GeoPoint 給地圖
   - **Phase B 寫立木**：每筆自動跑 `calcTreeMetrics`（材積、生物量、碳、CO₂）
   - 進度條每 5 筆更新一次，顯示「寫入樣區 N/M」「寫入立木 N/M」
6. 完成後綠色 banner 顯示三類統計：**成功 / 跳過（已存在）/ 失敗**，失敗清單可摺疊查看詳情（保留前 50 筆錯誤）

### 三、預設值（匯入產生的樣區與立木統一規格）

| 欄位 | 預設值 |
|---|---|
| `plot.shape` | `square`（方形）|
| `plot.area_m2` | `500`（0.05 公頃，林保署常用）|
| `plot.qaStatus` | `pending`（待 PI 審核）|
| `plot.notes` | `從 Excel 匯入 — sourceSheet: <sheet 名>` |
| `tree.qaStatus` | `pending` |
| `tree.statusCodeRaw` | 保留 Excel 原始狀態碼供審計 |
| 未知樹種的 `speciesSci` / `conservationGrade` | `null`（後續手動補）|

### 四、Idempotent 重跑

匯入失敗想重跑？直接再跑一次 wizard 即可：

- 已寫入的 plot.code 會偵測到 → 跳過
- 已寫入的 tree.treeCode 會偵測到 → 跳過
- 只補新增的部分 → 不會建重複資料

### 五、匯入後的事

1. **檢查地圖分頁**：所有匯入樣區應已出現在 OpenStreetMap 上（GeoPoint 自動換算）
2. **逐樣區審核**：到「待審核」分頁逐筆 ✓ 通過或 ⚠ 標記補正
3. **未知樹種補字典**：請 admin 在「設定 → 物種字典」（v2.7+ 預計新增）補齊 — 暫時可直接編輯每筆 tree 改正樹種
4. **無 GPS 樣區**：若 Excel 沒給絕對 X/Y、只有局部 X/Y → plot.locationTWD97 = null，地圖看不到。需 surveyor 帶手機現場補 GPS

> ⚠ **匯入前建議**：先建立**測試專案**跑一輪驗證流程沒問題，再對正式專案執行。已實機驗證：紙漿廠 19 樣區 / 930 立木 / 0 失敗 / 60-90 秒寫入完成。

> 📷 **圖位 14**：Excel 匯入 wizard 第 5 步 DRY-RUN 預覽區（顯示三個統計卡 + 紅色「執行真實匯入」按鈕 + payload 摺疊區）
> 📷 **圖位 15**：匯入進行中的進度條（顯示「寫入立木 N/M」+ 階段標籤）
> 📷 **圖位 16**：匯入完成後的綠色 banner（成功 / 跳過 / 失敗 三類統計卡）

### 五、第 5 步：審核 surveyor 提交的資料（QA/QC 主場）

切到「**待審核**」分頁（surveyor 提交資料後會看到 pending 數字）：

1. 看清單裡每筆 plot / tree / 各模組資料
2. 三選一：
   - **✓ 通過** → `qaStatus = verified`（綠色 badge）
   - **⚠ 標記** → `qaStatus = flagged`（橘色，附評論）→ surveyor 會看到「我被 flag」清單
   - **✕ 駁回** → `qaStatus = rejected`（紅色）→ surveyor 必須重做
3. 也可直接編輯小錯（例如打字錯誤）後通過

> 🆕 **v2.6.1 改善**：審核 sub-doc（立木 / 自然更新等）按 ✓ 後**只局部更新該 row 的 QA badge**，整頁不再閃爍 — 大批量審核（例：930 立木）時體感大幅改善。觸發狀態機 promote（例：全資料 verified → 自動 Lock）才會整頁重繪。
> 🆕 **v2.6.1 樣區清單卡片立木計數 chip**：審核任一立木後，「樣區」分頁卡片右上角的「🌳 N/M ⏳/✓」chip 會即時跳一格，不需切回樣區清單頁也能看到全 verified 的進度。詳見 [捌、五](#五v261-qa-ux-改善chip--局部更新)。

> 📷 **圖位 4**：待審核分頁清單（顯示 pending 計數、QA 按鈕、樣區/立木 row）

### 六、第 6 步：自動進入審查階段（全 verified 觸發）

當所有 plot + 所有子模組（含 v2.0+ 的地被/水保/野生動物/經濟收穫）都被你標記為 `verified`，系統會：

1. 自動把 `project.status` 從 `active` 升到 `review`
2. 自動 `Lock` 全專案資料（全員無法寫入）
3. Toast 通知：「🔍 全資料 verified — 專案已自動進入審查階段並鎖定」

如果之後你或別人 flag 任何一筆，系統會自動 Unlock 退回 `active`。

### 七、第 7 步：手動 Lock / Unlock（替代方案）

也可以不等全 verified，PI 直接「設定」分頁 →「Lock 專案」手動鎖。

### 八、第 8 步：匯出資料

切到「**匯出**」分頁：

- **Excel 全部**：一檔多 sheet（plot / tree / regen / understory / soilCons / wildlife / harvest）
- **CSV 各表**：每模組一檔，方便丟 R / Python / QGIS

> 📷 **圖位 5**：匯出分頁按鈕區

## 肆之三、admin 專案收尾（封存與永久刪除）

> 適用：案件結案後，admin 收尾把專案從作用中清單移除；實在不要的（測試專案、誤建）才永久刪除。
> ⚠ **角色限制**：本節操作**僅限 admin**。PI 沒有封存與永久刪除權限。

### 一、三步驟生命週期

```
作用中 (active)  ──📦 封存──▶  已封存 (archived) ──🗑 永久刪除──▶  徹底消失
                ◀──↺ 解封存──
```

| 動作 | 觸發位置 | 結果 |
|---|---|---|
| **封存** | 「我的專案」卡片右上「📦 封存」按鈕 | `archived=true` + `locked=true`（自動 Lock）；從預設清單移除（折疊到「📦 已封存」區） |
| **解封存** | 已封存卡片「↺ 解封存」按鈕 | `archived=false` + `locked=false`（自動 Unlock）；還原為作用中 |
| **永久刪除** | 已封存卡片「🗑 永久刪除」按鈕（**只在已封存才出現**）| 級聯刪除所有 plots / trees / regen / Storage 照片 + project doc — **無法復原** |

### 二、封存（admin）— 案件正常結案

**(一) 操作步驟**

1. 「我的專案」清單找到要結案的專案卡片
2. 點卡片右上「📦 封存」按鈕（amber 黃色）
3. 確認 dialog 顯示：
   - ✓ 所有資料保留在 Firebase 雲端（不會刪除）
   - ✓ 自動 Lock，沒人能再寫入
   - ✓ 從預設「我的專案」清單移除
4. 按「確定」→ Toast「已封存」+ 卡片移到頁面底部「📦 已封存（N）」區

**(二) 已封存區**

- 「我的專案」清單最下方有「📦 已封存（N）」標題
- 預設折疊（hidden），點「顯示」展開
- 展開後可看到所有已封存卡片，仍可點進去**唯讀檢視**全部資料
- 仍可正常匯出 Excel / CSV

### 三、解封存（admin）— 還原為作用中

需要重新編輯已封存專案？點「↺ 解封存」按鈕（藍色）：
- `archived=false` + `locked=false`
- 卡片回到「作用中」清單
- PI 又可以寫入了
- 若還想保持「不可改」狀態 → admin 可請 PI 手動 Lock

### 四、永久刪除（admin）— 三道安全閘門

> ⚠ **此操作無法復原**。所有 plots / trees / regen / understory / soilCons / wildlife / harvest 文件 + Storage 照片 + project 本身全部消失。

**(一) 前置條件**

1. **必須先封存**（`archived=true`）— 系統強制至少二步驟思考
2. **建議先匯出資料備份**（XLSX / CSV）— 留存原始資料防後悔
3. **PI 已手動 Unlock**（若是 v1.5 legacy locked 或自動 Lock）— 否則 Firestore Rules `NOT isLocked` 會擋級聯刪除 ⚠

**(二) 操作步驟**

1. 「📦 已封存」展開區找到要刪除的卡片
2. 點「🗑 永久刪除」按鈕（紅色，**只在已封存卡片才出現**）
3. 客製確認 modal 跳出，必須完成：
   - ☑ 勾選「我已從『匯出』分頁下載此專案的資料備份（XLSX / CSV）」
   - ✏ 輸入專案代碼（例：`CHP-TTFB-2026-001`）— 防誤刪
4. 按「永久刪除」→ Toast「刪除中...」（最多 30 秒）
5. 系統依序刪除：
   - (a) 所有 trees（含個體照片）
   - (b) 所有 regeneration / understory / soilCons / wildlife / harvest
   - (c) 所有 plots + Storage 照片（用 `listAll` 一網打盡 prefix）
   - (d) project doc
6. 完成後 Toast「已刪除：N 樣區 / M 立木 / K 更新 / J 照片」
7. 自動跳回首頁

**(三) 已知 UX 痛點**

- 若專案 `isLocked=true`（自動 Lock 或 legacy Lock）→ Firestore Rules 會擋級聯刪除 → 需請 PI 先 Unlock 才能刪
- v2.6.x backlog 🅴 已記錄「`deleteProjectCascade` 啟動時自動 force-unlock」改善（未排程）

### 五、admin 收尾 SOP 建議

```
案件結束
  ├─ ① PI 確認所有資料已匯出（Excel + CSV）
  ├─ ② PI 通知 admin 結案
  ├─ ③ admin 點「📦 封存」按鈕 → 已封存
  ├─ ④（一年後或測試專案）admin 確認 PI 同意永久刪除
  ├─ ⑤ admin 再匯出一次最終備份（建議異地保存）
  ├─ ⑥（若 isLocked）PI 手動 Unlock
  └─ ⑦ admin 點「🗑 永久刪除」按鈕 → 完成 3 道閘門
```

> 📷 **圖位 17**：「我的專案」清單顯示「作用中」與「📦 已封存」兩區（含 admin 三按鈕：📦 封存 / ↺ 解封存 / 🗑 永久刪除）
> 📷 **圖位 18**：永久刪除確認 modal（勾選「已備份」+ 輸入專案代碼）

## 伍、Surveyor 操作流程（現場調查）

### 一、第 1 步：登入後自動跳到指派專案

如果你只被指派一個專案，登入後會自動進入該專案的「樣區」分頁。

如果有多個專案，先在「我的專案」點選。

### 二、第 2 步：建立或開啟樣區

**(一) 情境 A：PI 已建空殼樣區並指派給你**

1. 在「樣區」分頁找到你被指派的卡片（顯示「指派給：我」）
2. 點進去 → 「樣區資訊」分頁 → 「編輯樣區資訊」
3. 抓 GPS（按「📍 取得 GPS」按鈕，需允許瀏覽器位置權限）
4. 確認林班、面積、形狀、設置日期
5. 儲存

**(二) 情境 B：自由加新樣區**

1. 「樣區」分頁右上「＋ 新樣區」
2. 同樣抓 GPS、選面積（從 PI 設的允許值挑）
3. 儲存

> 📷 **圖位 6**：樣區編輯表單（顯示 GPS 按鈕 + TWD97 即時換算 + 精度顯示）
> 📷 **圖位 7**：「樣區無 GPS」橘色警示（v2.3.9 加的「⚠ 尚未設定 GPS 座標 [✎ 點此設定]」）

### 三、第 3 步：填立木 / 自然更新等子模組

進入樣區後有 sub-tab：🌳 立木調查 / 🌱 自然更新 / 🌿 地被植物 / ⛰️ 水土保持 / 🦌 野生動物 / 🌰 經濟收穫 / 📋 樣區資訊

依 PI 啟用的模組顯示。

**🌳 立木：**
1. 「＋ 新立木」
2. 輸入樣木號碼（系統自動帶下一號）
3. 樹種（自動完成下拉，含 100 種台灣樹種；保育類自動顯示警示）
4. DBH（cm）+ 樹高（m）→ 系統即時算斷面積、材積、生物量、碳、CO₂
5. **🆕 v2.5 立木個體位置 X/Y**（選填）— 詳見下方
6. 活力（健康/衰弱/枯立/倒伏）+ 病蟲害（複選）+ 標記方式
7. 拍照（樹皮/葉/花果/整體外觀）
8. 儲存

#### v2.5 立木個體 X/Y 座標（樣區內定位）

> 為什麼要量？立木個體位置 = 樣區內局部 X/Y（公尺）。系統會自動換算絕對 TWD97 + WGS84 GeoPoint 給地圖與 GIS 套疊用。林業保育署永久樣區、固定樣區複測、競爭分析、樣區內空間異質性研究都會用到。

**(一) 兩種原點模式（PI 在「設計」分頁設定）**

| 模式 | 原點位置 | X/Y 正負號 | 適用 |
|---|---|---|---|
| **中心點**（`center`，預設） | 樣區中心 | X：東 + 西 −；Y：北 + 南 − | 林保署永久樣區、4 象限皮尺 |
| **左下角**（`corner`） | 樣區西南角 | X / Y 皆為正 | 一般方形樣區、由角落開始量 |

> ⚠ surveyor 自己**不能改**原點模式，PI 設好之後立木表單會自動帶該模式的提示文字（例：「皮尺距樣區中心點的偏移量（東向 X / 北向 Y，可正可負）」）。

**(二) 操作步驟**

1. 立木表單滾到「📍 立木位置（中心點原點）」藍色區塊（標題會依模式變「中心點」或「左下角」）
2. 輸入 X / Y（公尺，可帶 2 位小數，例：`-3.50`、`12.80`）
3. 表單下方即時顯示換算結果：
   - 絕對 TWD97：`X=300150.50 m, Y=2580436.20 m`
   - WGS84：`121.123456, 23.654321`
4. 系統會把 `localX_m` / `localY_m` / `locationTWD97` / `location`（GeoPoint）四個欄位一起存進 tree 文件

**(三) 樣區未設 GPS 時的行為**

如果樣區的 `locationTWD97` 還是空的（surveyor 還沒抓樣區中心 GPS），表單會顯示：

```
⚠ 樣區尚未設定 GPS 座標 — 無法換算絕對位置
```

→ 立木 X/Y 仍可填存，但只是相對值；之後 PI / surveyor 補上樣區 GPS 後系統會自動補算 absolute（**目前實作為儲存當下計算，後續補 GPS 不會回填舊立木的絕對座標 — 這是已知限制，請先抓樣區 GPS 再填立木 X/Y**）。

**(四) 散布圖視覺化（v2.5.1 規劃中）**

未來版本會在樣區詳情加一個「分布」tab，把 X/Y 畫成 Canvas 散布圖。目前只能透過匯出 Excel / CSV 自己用 R / Python / QGIS 畫。

> 📷 **圖位 8**：立木表單（顯示樹種 autocomplete + DBH/H + 即時計算區 + 病蟲害 checkbox）
> 📷 **圖位 8b**：立木表單 X/Y 區塊（藍色背景，標題顯示「📍 立木位置（中心點原點）」+ X/Y 輸入框 + 即時換算 TWD97 / WGS84 顯示）

**🌱 自然更新：** 簡化版，物種 + 棵樹 + 高度級距

**🌿 地被植物：** 5 點樣方法（北/東/南/西/中央），物種覆蓋度估值；入侵種橘色標示

**⛰️ 水土保持：** 5 點定點觀測；颱風/豪雨後可加事件紀錄；自動帶上次同點位代表照供比對

**🦌 野生動物：** 4 種方法（直接目擊/痕跡/自動相機/鳴聲）；保育類 I/II/III 級紅/橘/黃色階；含與 plot 中心 Haversine 距離

**🌰 經濟收穫（土肉桂為首）：** 綁立木個體；treeStatusAfter=砍除根除自動同步 tree.vitality；鮮重→乾重→自動算 tCO₂e 扣減

### 四、第 4 步：拍照（所有模組通用）

1. 點「上傳照片」（每個模組都有）
2. 選擇相簿照片或當場拍 — 自動壓縮到 ≤5MB
3. 上傳中會顯示進度，完成顯示縮圖
4. 點縮圖可預覽 / 移除

### 五、第 5 步：離線使用

如果現場無訊號：
- 平臺仍可正常開啟（PWA 已快取）
- 填的資料暫存本地 IndexedDB
- 上方會出現「📵 離線模式 — 資料已暫存，回連時自動同步」橘色 banner
- 回到有訊號處自動上傳，無需任何操作

> ⚠️ **離線注意**：照片上傳需網路（Firestore 文字資料可離線，但 Firebase Storage 需網路）。建議離線時先填表不傳照片，回連後再補照片。

### 六、第 6 步：看「我被 flag」清單（被 PI 退回時）

如果你提交的資料被 PI 標記為 `flagged` 或 `rejected`：

1. 進入專案後**會出現「我被 flag」分頁**（含紅色數字）
2. 點進去看 PI 的評論
3. 修正後重新儲存 → 自動把 `qaStatus` 改回 `pending` 等 PI 重審

## 陸、Reviewer 操作流程（外部審查視角）

### 一、第 1 步：被邀後登入

PI 邀請你後，你登入會看到該專案。

### 二、第 2 步：你看得到的分頁

reviewer 進入專案後可看到 **5 個分頁**：

| 分頁 | 內容 |
|---|---|
| 🌳 樣區 | 全專案樣區清單，可點進每個樣區看立木與各模組資料；**調查者欄位顯示匿名「調查員 A / B / C…」** |

| 📊 儀表板 | 進度卡 + KPI + 直徑分布 + 樹種 IV + 健康狀態圖 |
| 🗺️ 地圖 | 全樣區 GPS 位置（OpenStreetMap 底圖） |
| 📤 匯出 | Excel 全部 / 各模組 CSV — **reviewer 可自行下載做進階分析** |

reviewer **看不到**：「待審核」「我被 flag」「設定」三個分頁。

### 三、第 3 步：儀表板 — 整體資料品質與分析

- **進度卡**（跨 2 欄頂部）：百分比 + 綠色進度條 + verified/total + 6 色專案狀態 badge + 待審/flagged/rejected breakdown
- **KPI 卡**（7 格）：樣區數、總調查面積、立木總數、總材積、總斷面積、總碳蓄積、總 CO₂ 當量
- **直徑分布**：DBH 5cm 級的 bar chart
- **樹種重要值（IV）**：前 10 種水平 bar；公式 = (相對密度 + 相對優勢度) / 2 × 100
- **健康狀態**：donut（健康/衰弱/枯立/倒伏 4 色）

### 四、第 4 步：地圖 — 樣區位置

地圖顯示所有樣區 marker（OpenStreetMap 底圖），點 marker 可看樣區資訊卡。

> ⚠️ **v2.4.0 注意事項**：reviewer 看到的座標**與 PI 一樣精確**（沒有座標模糊化）。「reviewer 地圖座標模糊到 100m grid」是 v3 規劃中功能，目前未實作。**敏感物種（如保育類 I 級）點位若不適合外流，請 PI 在邀請 reviewer 前評估。**

### 五、第 5 步：匯出資料做進階分析

reviewer **自己**可以在「匯出」分頁下載：

- **Excel 全部** — 一檔多 sheet（plot / tree / regen / understory / soilCons / wildlife / harvest）
- **CSV 各表** — 每模組單檔，可餵 R / Python / QGIS

> 📷 **圖位 9**：reviewer 看到的儀表板（進度卡 + 7 個 KPI + 三張圖；樣區清單上的調查者欄位匿名為 A/B/C…）
> 📷 **圖位 10**：reviewer 看到的地圖（v2.4.0 顯示真實座標，與 PI 視角相同）

## 捌、QA 狀態機與 Lock 機制

### 一、4 個 QA 狀態（資料層）

| 狀態 | Badge | 意義 | 誰能設 |
|---|---|---|---|
| **pending** 待審核 | 灰 | 預設值，surveyor 剛提交 | 系統自動 |
| **verified** 已通過 | 綠 | PI 確認資料無誤 | PI |
| **flagged** 待修正 | 橘 | PI 有疑問，附評論 | PI |
| **rejected** 已駁回 | 紅 | PI 認為錯誤需重做 | PI |

### 二、6 個專案狀態（v2.3 階段 2 加）

```
created → planning → active → review → verified → archived
   ↑─────────────────────┘ (PI 退回 flag 時自動)
```

| 狀態 | 觸發 | 含意 |
|---|---|---|
| **created** | admin 剛建立 | 等 PI 設方法學 |
| **planning** | PI 第一次儲存方法學 | 可加樣區了 |
| **active** | 有任何 plot/tree 資料 | 收集中 |
| **review** | 全資料 verified | 自動 Lock + 進審查 |
| **verified** | reviewer 通過審查（v3 才實作） | — |
| **archived** | PI 手動歸檔 | 完整唯讀 |

### 三、Lock 邏輯

| 觸發 | 結果 | 自動 Unlock 條件 |
|---|---|---|
| 全資料 verified（自動）| `locked=true` + `autoLockReason=all-verified` | PI flag 任何一筆即自動 Unlock |
| PI 手動「Lock 專案」 | `locked=true` + `autoLockReason=manual` | PI 手動「Unlock」 |
| Migration legacy 鎖（v1.5 既有 Lock）| `autoLockReason=legacy` | PI 手動「Unlock」 |

### 四、QA 推進 — 狀態機自動 promote / demote

當 PI 在「待審核」分頁標記 sub-doc（立木 / 自然更新 / 各模組）：

| 觸發條件 | 結果 |
|---|---|
| **plot 內某 sub-doc 從 pending 變 verified** + 該 plot 的所有 sub-doc 都 verified | plot.qaStatus 自動 promote 到 verified |
| **plot.qaStatus 變 verified** + 專案內所有 plot 都 verified | project.status 自動 promote 到 review + 自動 Lock |
| **plot 內任一 sub-doc 從 verified 變 flagged/rejected** | plot.qaStatus 自動 demote 到 flagged + project 自動 Unlock + status 退回 active |

### 五、v2.6.1 QA UX 改善（chip + 局部更新）

> 痛點：v2.3.1 引入的「全頁 reroute」對審 sub-doc 場景是大砲打蚊子 — 紙漿廠 949 筆審核每筆閃 1 次受不了。v2.6.1 把 sub-doc 改為「局部更新」，狀態機 promote/demote 才走全頁重繪。

**(一) markQA 行為分流**

| 場景 | v2.6.1 行為 | 為什麼 |
|---|---|---|
| 改 plot 本身的 QA | **整頁 reroute** | lock banner / status badge / 進度卡都需要更新 |
| 改 sub-doc 且**未觸發**狀態機 promote/demote | **局部更新 row badge** | 該 row 的 ✓/⏳ icon 直接換 outerHTML |
| 改 sub-doc 且**觸發**狀態機 promote/demote | **整頁 reroute** | plot.qaStatus 也變了，全頁要重繪 |

實作上每個 QA cell 都帶 `data-qa-cell-id` 屬性，markQA 結束時用 `document.querySelector` 抓到 cell → 替換 badge HTML，不重繪整個 view。

**(二) 樣區清單卡片立木計數 chip**

樣區清單每張卡片右上角會顯示一個 chip，格式：

```
🌳 N/M ⏳    （部分 verified — 橘）
🌳 N/M ✓    （全 verified — 綠）
🌳 0/0      （沒立木或還沒開始 — 灰）
```

- **N**：該 plot 內 verified 的立木數
- **M**：該 plot 內 trees collection 總筆數
- 顏色：綠（N=M）/ 橘（0<N<M）/ 灰（M=0）
- chip 隨「按 ✓ 立木」即時跳一格 — **不用切回樣區清單頁也會更新**
- 實作：markQA 寫入後 dispatch `mrv:qa-changed` CustomEvent，window listener 找到對應 chip（`[data-trees-chip-plot=...]`）依 dataset 現值 ±1 重建

**(三) 為什麼 chip 不能用 onSnapshot 訂閱？**

實務上踩過坑：
- Firestore Rules 對 `collectionGroup('trees')` 拿不到 `projectId` path 變數 → 必然 permission-denied
- 改用 sub-collection query（每個 plot 一次 getDocs）+ Promise.all 平行 — 較簡單也不違反 rules
- onSnapshot 訂閱 plot 對 sub-doc 改動「無感」 — 必須額外 broadcast 機制（CustomEvent）

**(四) chip 目前只統計 trees**

v2.6.1 為了快速 ship 只統計 🌳 立木，其他模組（regen / understory / soilCons / wildlife / harvest）尚未統計。**v2.6.x backlog 🅳** 已記錄「依 methodology.modules 動態擴展 chip」。

> 📷 **圖位 20**：樣區清單卡片右上角 chip（橘色「🌳 12/67 ⏳」部分 verified 範例）

## 拾、常見問題（FAQ）

### 一、登入相關

**Q：被邀請後登入仍看不到專案？**
A：請先用 Google **登入過一次**讓系統建你的 user 文件，再請 PI 重新加你。

**Q：Google 登入彈窗被擋？**
A：v1.5.6 已加 popup fallback。如仍失敗，瀏覽器地址列點 🔒 圖示 → 允許彈出視窗。

### 二、現場調查相關

**Q：GPS 抓不到？**
A：① 瀏覽器允許位置權限 ② iPhone 設定 → 隱私 → 定位服務 → Safari → 允許 ③ 室內訊號弱，建議走出開闊處 ④ 平臺顯示「精度 ±100m」太大時可重抓

**Q：照片上傳很慢 / 失敗？**
A：① 檢查訊號 ② 單張限 5MB（自動壓縮，但原圖 >20MB 可能出錯）③ 離線時不傳照片，回連再補

**Q：填到一半換手機接著填？**
A：資料已自動雲端同步，換手機登入同帳號就接得起來。

**Q：填表時跳出「資料已 Lock」？**
A：① 全資料 verified 後系統會自動 Lock — 請 PI 看是否有 flag 退回 ② PI 手動 Lock — 跟 PI 確認是否暫時 Unlock 給你補資料

### 三、QA / 審核相關

**Q：PI 為什麼退我的立木資料？**
A：到「我被 flag」分頁看 PI 的評論。常見原因：DBH 與樹高比例不合理、樹種誤判、座標離樣區太遠、照片模糊。

**Q：Surveyor 看得到別的調查員資料嗎？**
A：可看（透明度設計），但**不能改**。reviewer 看到的調查者名字會匿名化為 A/B/C…

### 四、技術相關

**Q：剛改的功能沒看到？**
A：PWA 有 cache。**強制重整**：桌機 `Ctrl + Shift + R` / 手機砍 App 後重開。

**Q：Excel 匯出檔在 R 內讀中文亂碼？**
A：用 `readxl::read_excel()` 而非 `read.csv` — 前者預設 UTF-8。

**Q：可以匯入既有 Excel 樣區資料嗎？**
A：✅ **v2.6.0 起已支援**（2026-04-27 deploy 上線）。詳見 [肆之二](#肆之二替代方案--從-excel-批次匯入既有資料v260-新功能)。實機驗證：紙漿廠 19 樣區 / 930 立木 / 0 失敗 / 60-90 秒匯入完。

**Q：立木個體位置 X/Y 是必填嗎？**
A：**選填**。但若想要做樣區內競爭分析、空間異質性研究、林保署永久樣區複測對位 → 強烈建議填。先請樣區設好 GPS（中心點或左下角）再填立木 X/Y，否則絕對座標會缺。

**Q：填了立木 X/Y 但樣區還沒 GPS 怎麼辦？**
A：可以存（locationTWD97 / location 會是 null），但**目前實作不會在補 GPS 後回填**舊立木的絕對座標 — 這是已知限制（v3 會改）。建議先抓樣區 GPS 再填立木 X/Y。

**Q：審核時整頁閃個不停受不了？**
A：v2.6.1 已改 — 審 sub-doc（立木等）只局部更新該 row badge，整頁不再閃；只有觸發狀態機 promote（plot 全 verified 自動 Lock）才整頁重繪。請強制重整一次（Ctrl+Shift+R）拿到最新版。

**Q：可以永久刪除測試專案嗎？**
A：可以，但只有 **admin** 能做，且必須先「📦 封存」才看得到「🗑 永久刪除」按鈕。詳見 [肆之三](#肆之三admin-專案收尾封存與永久刪除)。三道安全閘門：① 必先封存、② 勾選「已備份」、③ 輸入專案代碼確認。

**Q：永久刪除按下去卡住沒反應？**
A：很可能專案還在 Lock 狀態（自動 Lock 或 legacy Lock）→ Firestore Rules 擋級聯刪除。請 PI 先到「設定」分頁手動「Unlock」再刪。v2.6.x backlog 🅴 已記錄「自動 force-unlock」改善。

## 拾壹、附錄

### 一、TWD97 ↔ WGS84 坐標

平臺內部存兩種坐標：
- `location`（GeoPoint）：WGS84，給地圖顯示用
- `locationTWD97`：`{x, y}`，給 GIS 套疊用（與台灣公部門圖層一致，TM2 中央經線 121°）

東部離島（綠島、蘭嶼、東部離岸）建議使用 TM2 中央經線 119° 系統，平臺目前統一用 121°，誤差可接受但需註明。

### 二、24 樹種異速生長式

立木調查的材積/碳量計算覆蓋以下樹種（其他樹種自動 fallback 到「其它針」或「其他闊」公式）：

**針葉樹**：紅檜、台灣杉、香杉、二葉松、琉球松、鐵杉、冷杉、雲杉、台灣肖楠、台灣黃杉
**闊葉樹**：樟樹、楠木類、櫧櫟類、相思樹、大葉桃花心木、光腊樹、銀合歡、泡桐、楓香、台灣赤楊、其他闊
**Fallback**：其它針 / 其他闊

碳計算鏈：`V (m³) → 生物量 (kg) [×WD×BEF] → 碳 (kg) [×CF=0.47] → CO₂ (kg) [×44/12]`

### 三、6 模組欄位完整 schema

請參考同層的 `資料schema.md`（含 Firestore collection 結構與每欄位型別、必填、說明）。

### 四、權限與 Security Rules 細節

請參考同層的 `Firestore-security-rules.md`。
