> 上一版：v1.0（隱含設計，無正式文件）
> 拍板紀錄：見「捌、決策紀錄」段

## 壹、總覽

本系統以**三層權限模型**搭配**五種角色**，建立科學監測資料的「**雙重複核（double check）+ 鎖定後不可改**」品質保證機制，符合 MRV（測量、報告、查證）與 TACCC（透明、一致、可比、完整、準確）原則。

```
┌─────────────────────────────────────────────────────┐
│ 第一層：系統層（/users/{uid}.systemRole）             │
│   admin（系統管理者） / member（一般使用者）           │
└──────────────────────┬──────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────┐
│ 第二層：專案層（/projects/{pid}.members[uid] = role）  │
│   pi / dataManager / surveyor / reviewer            │
└──────────────────────┬──────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────┐
│ 第三層：資料擁有權（/.../doc.createdBy == uid）       │
│   surveyor 僅能編輯 createdBy == 自己 的資料           │
└─────────────────────────────────────────────────────┘
```

## 貳、五種角色職責

### 一、系統管理者（admin）— 資料庫管家

**真實對應**：系統建置者、IT 助理

| 能做 | 不能做 |
|------|--------|
| ✅ 建立空殼專案（名稱、代碼、描述、指派 PI） | ❌ 改方法學 |
| ✅ 跨專案管理 user 帳號（停用、刪除帳號） | ❌ 寫入任何專案資料 |
| ✅ 刪除空殼專案（plots 子集合為空時） | ❌ 邀請成員（PI 才能） |
| ✅ **唯讀**所有專案資料（系統審計用） | ❌ 修改 QA 狀態 / Lock |

> 💡 **設計原則**：admin 是「**facility manager**」，負責開闢空間 + 系統審計；不參與任何專案的科學決策與資料生產。

### 二、計畫主持人（pi）— 科學負責人

**真實對應**：教授、計畫主持人、林業署科員

| 能做 | 不能做 |
|------|--------|
| ✅ 設計方法學（樣區規格、目標數、必填欄位、調查模組開關） | ❌ 跨專案操作 |
| ✅ 邀請/移除成員、指派角色 | ❌ 建新專案（admin 才能） |
| ✅ 全資料讀寫 + 刪除 | ❌ 跨專案看別人的資料 |
| ✅ Lock / Unlock 資料 | |
| ✅ QA/QC 爭議的最終仲裁 | |
| ✅ 匯出 Excel / CSV | |

### 三、資料管理員（dataManager）— QA/QC 把關者

**真實對應**：研究助理（office-based）、計畫專任助理

| 能做 | 不能做 |
|------|--------|
| ✅ **複核（double check）所有 surveyor 提交的資料** | ❌ 改方法學 |
| ✅ 標記 `qaStatus`：verified / flagged / rejected | ❌ 邀請成員 |
| ✅ 寫 QA 評論（v2）、退回修正、追蹤復檢 | ❌ Lock / Unlock |
| ✅ 編輯任何資料（修小錯，留 audit log） | ❌ 刪除專案 |
| ✅ 看「待審核」清單分頁（主場） | |
| ✅ 匯出 | |
| ⚠️ **不應親自收集資料**（系統不阻擋，但 SOP 規範） | |

> 💡 **Double-check 原則**：dataManager 與 surveyor **必須是不同的人**，且 dataManager **不應複核自己收集的資料**。系統不強制（小計畫常需兼任），但專案 SOP 文件應寫明此原則。

### 四、野外調查員（surveyor）— 資料生產者

**真實對應**：學生、研究生、約聘調查員

| 能做 | 不能做 |
|------|--------|
| ✅ 野外資料蒐集（樣區、立木、自然更新） | ❌ 編輯他人資料 |
| ✅ 編輯**自己**建立的資料（資料未 Lock 前） | ❌ 標記 QA 狀態（dataManager 才能） |
| ✅ 看全專案儀表板與地圖（**透明度**） | ❌ Lock / Unlock |
| ✅ 看全專案樣區清單（含他人的，含 QA 狀態） | ❌ 邀請成員 |
| ✅ 看「我被 flag」清單 + 復檢 | ❌ 改方法學 |
| ✅ 刪除自己未提交的草稿 | |
| ✅ 看自己樣區的調查者真名 | |

> 💡 為何 surveyor 看得到他人原始資料？基於**透明度與互相學習**。如果擔心「互相影響」（confirmation bias），請在 SOP 中規範「填表前不看他人資料」即可。

### 五、審查委員（reviewer）— 唯讀外部觀察者

**真實對應**：補助方代表、外部審查委員、跨單位顧問

| 能做 | 不能做 |
|------|--------|
| ✅ 看儀表板、匯出 Excel/CSV | ❌ 任何寫入操作 |
| ✅ 看樣區清單（含 QA 狀態） | ❌ 看調查者真名（匿名為 A/B/C） |
| ✅ 看地圖（**去識別化**：座標精度降到 100m） | ❌ 進入「設計」「設定」「待審核」分頁 |
| ✅ 看 QA 狀態統計 | |

## 參、工作流程（四階段 Double-Check）

### 一、Setup 階段（admin → pi 交接）

```
admin 開 App → 「+ 新專案」→ 填名稱+代碼+描述+指派 PI 的 email
       ↓
admin 完成「開闢空間」→ 退出該專案（不再參與）
       ↓
pi 收到邀請通知 → 進入空殼專案 → 「設計」分頁設定方法學
       ↓
pi 「設定」分頁邀請 dataManager + surveyors
```

### 二、資料蒐集階段（surveyor 主場）

```
surveyor 開 App → 自動跳到唯一指派專案 → 看到「樣區」分頁
       ↓
+ 新樣區（系統自動套用 PI 設定的方法學樣板）
       ├─ 若已達 targetPlotCount → 警示「已達目標數 50 個，是否繼續？」
       └─ 抓 GPS、填表、可離線
       ↓
回收訊區 → 自動同步 → 資料寫入 Firestore，qaStatus = "pending"
```

### 三、QA/QC 階段（dataManager 主場）— **Double Check 核心**

```
dataManager 開 App → 進入專案 → 「待審核」分頁（filter qaStatus=pending）
       ↓
逐筆 review：
  ├─ 數值合理 + 完整 → qaStatus = "verified" ✅（綠色 badge）
  ├─ 有疑問 → qaStatus = "flagged" + 寫評論（v2）→ 通知 surveyor 復檢
  └─ 嚴重錯誤（座標離譜、樹種誤判）→ qaStatus = "rejected"（紅色）
       ↓
surveyor 看到自己被 flag/rejected 清單 → 修正 → 重新提交（qaStatus 回 pending）
       ↓
dataManager 復檢 → verified
       ↓
所有筆都 verified → pi 知道可以進入下階段
```

### 四、報告階段（pi + reviewer）

```
pi 「設定」分頁按 「Lock」→ 全專案資料凍結（包含 dataManager 也不能改）
       ↓
pi 匯出 Excel → 跑 R/Python 分析 → 寫報告
       ↓
reviewer 開唯讀儀表板 → 看摘要 → 申請特定資料時 pi 匯出給他
```

### 五、Audit Log（v2 加）

每次 dataManager / pi 編輯他人資料，自動記錄：
- 誰改的（uid）
- 什麼時候（timestamp）
- 改了什麼欄位（diff）
- 為什麼（強制填寫 reason）

## 伍、Schema 增補（v1.5 必要欄位）

### 一、`/users/{uid}` 增補

```js
{
  // 既有
  email, displayName, createdAt,
  // 改名（globalRole → systemRole）
  systemRole: 'admin' | 'member'  // default 'member'
}
```

### 二、`/projects/{pid}` 增補

```js
{
  // 既有
  code, name, description, members, createdBy, createdAt,
  
  // 新增 v1.5
  memberUids: ['uid1', 'uid2', ...],  // 給 onSnapshot 查詢用
  pi: 'uid',                          // 主 PI（members 中也會有）
  
  methodology: {
    targetPlotCount: 50,              // 目標樣區數
    plotShape: 'circle',              // 'circle' | 'square'
    plotAreaOptions: [500, 1000],     // 允許的面積選項
    required: {                       // 必填欄位開關（true=強制）
      photos: false,
      branchHeight: false,
      pestSymptoms: false
    },
    modules: {                        // 啟用的調查模組
      plot: true,
      tree: true,
      regeneration: true,
      // v2 加入:
      understory: false,
      soil: false,
      disturbance: false
    },
    description: '方法學說明...'
  },
  
  locked: false,                      // 鎖定狀態
  lockedAt: null,                     // 鎖定時間
  lockedBy: null                      // 鎖定者 uid
}
```

### 三、`/projects/{pid}/plots/{plotId}`（與 trees, regeneration）增補

```js
{
  // 既有
  ...所有原有欄位,
  
  // 新增 v1.5
  qaStatus: 'pending' | 'verified' | 'flagged' | 'rejected',  // default 'pending'
  qaMarkedBy: 'uid',                  // QA 標記者
  qaMarkedAt: timestamp,              // QA 標記時間
  qaComment: '評論文字'                // v1.5 inline；v2 改為子集合
}
```

## 陸、Security Rules 規則摘要

```
function isSystemAdmin() {
  return get(/users/{uid}).data.systemRole == 'admin';
}
function isPi(pid)        { return projects/{pid}.members[uid] == 'pi'; }
function isDataManager(pid) { return projects/{pid}.members[uid] == 'dataManager'; }
function isSurveyor(pid)  { return projects/{pid}.members[uid] == 'surveyor'; }
function isReviewer(pid)  { return projects/{pid}.members[uid] == 'reviewer'; }
function isMember(pid)    { return members[uid] != null; }
function isLocked(pid)    { return projects/{pid}.locked == true; }

// /projects/{pid}
allow create: only systemAdmin
allow update: only pi (含改 methodology, members)，但 lock 後限改 lock 欄位
allow delete: systemAdmin AND no plots
allow read:   isMember(pid) OR isSystemAdmin()  // admin 可讀全部

// /projects/{pid}/plots/{plotId}
allow create: pi OR dataManager OR surveyor，AND NOT isLocked
allow read:   isMember(pid) OR isSystemAdmin()
allow update: 
  - 一般欄位：pi OR dataManager OR (surveyor AND isOwner)，AND NOT isLocked
  - QA 欄位（qaStatus, qaMarkedBy, qaMarkedAt, qaComment）：pi OR dataManager only
allow delete: pi OR (surveyor AND isOwner AND qaStatus=='pending')

// 同樣模式套用到 trees, regeneration
```
